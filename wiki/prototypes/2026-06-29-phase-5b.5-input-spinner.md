# Prototype — 2026-06-29 — Phase 5b.5: multi-line input + status spinner

## Observation

Cycle 5b.5 addresses two specific gaps from the 5b walkthrough:

1. **Submit-signal ambiguity.** The 5b code used a single-line `prompt` for the first line of the move, then looped with `prompt('')` for additional lines until a blank line. The player couldn't tell when their move was sent: was one line enough, or did they need to keep typing? The 5b walkthrough revealed this concretely.

2. **No status during the LLM wait.** 5b removed the previous comedic-interlude during the LLM call (it was noise). But that left a 10-30s gap with nothing on screen — the player didn't know the system was doing anything.

The fixes:

### Multi-line input by default

The new `reader.promptMove(headerQuestion)` method does the conventional multi-line pattern: the header is printed once, then `  > ` is the prompt for each line, and a blank line ends the move. The first line of the move is captured directly via `reader.prompt('  Your move (or \`a\` for an advisor): ')` (the advisor shortcut), then subsequent lines are read with the `> ` prompt until blank. Either single-line or multi-line moves are valid.

The player experience:
- Type your move
- Press enter to start a new line (continues the move)
- Press enter on a blank line to submit
- The cursor stays at `  > ` between lines, so the player knows the move is being built

For the advisor shortcut: type `a` on the first line to enter advisor mode. The advisor is consulted, then the move is captured via `promptMove`.

### Status spinner during the LLM call

A new `withSpinner(reader, message, fn)` helper:

- **Piped mode** (non-TTY): prints a single static line `  Interpreting your move\n` and runs the function. No animation. Used for tests.
- **TTY mode**: prints a pendulum spinner that animates the message. The message text is always intact (the dot oscillates in the trailing space region, never overlapping the message). The spinner uses `\x1b[1A` (cursor up) + `\r` (carriage return) + the new frame to update in place. When the function returns, the spinner line is cleared (overwritten with spaces) and a newline is added, so the next turn's prose appears cleanly below.

The 8 frames are a pendulum: dot positions `[0, 1, 2, 3, 4, 3, 2, 1]`, repeated. Each frame is the same length (message + 4 trailing dot-position spaces), so the spinner is stable. The tick interval is 800ms, which feels calm — not anxious, not sluggish.

## What the player experience looks like

After pressing enter on a blank line, the player sees:
```
  Your move (or `a` for an advisor): We will convene a 60-day review.
  > 
  Interpreting your move ·    
```

Then the dot oscillates through the trailing spaces (800ms per position). When the LLM returns, the spinner line is cleared and the next turn's prose appears:
```
  
  ─── Turn 2 ───
  ...
```

## Design notes

**Why the message text is intact in every frame.** The first version of the spinner placed the dot at the end of `message.length + dotPos`, which inserted the dot *into* the message text (e.g., "Test messa·ge" instead of "Test message"). The fix: pad the message + 4 trailing spaces to a fixed length, then place the dot at a position that's always in the trailing region. The message is never broken.

**Why pendulum instead of rotation.** A 4-state rotation (clockwise, like ⠋ ⠙ ⠹ ⠸) requires Unicode braille characters that may not render in all terminals. A 3-state rotation (`·  ·  ·`) is sparse and doesn't feel like a continuous animation. The 8-state pendulum (forward, then back) is text-only, uses a single `·` character, and the back-and-forth motion reads as "thinking" without requiring Unicode or fancy ASCII.

**Why 800ms per tick.** Slower than 500ms (which feels anxious) and faster than 1.5s (which feels broken). 800ms is in the calm-and-informative range. The full 8-frame cycle takes 6.4s, so for a 30s LLM call, the player sees ~5 full cycles. That's enough motion to feel "the system is doing something" without being a distraction.

**Why the spinner line is cleared, not left in place.** If the spinner stayed, the next turn's prose would appear above the spinner (where the readline cursor was) or below it (offset by one line). Either way, the visual would be wrong. Clearing the line and printing a blank line gives the next turn a clean canvas. The total visual transition: `Interpreting your move ·    ` → `  ` → next turn's prose.

**Why the advisor shortcut uses a single line.** The advisor flow is a one-shot action: "I want to consult X, then write my move." It's not multi-line because the advisor selection is a number, not a paragraph. The single-line prompt keeps the advisor flow distinct from the multi-line move flow. After the advisor is consulted, the move goes through `promptMove` (multi-line) like normal.
