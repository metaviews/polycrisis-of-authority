## Interpretation grammar

The grammar takes the player's free-text policy input and produces state-vector deltas on the six state axes. It is the central mechanism of the simulation. Per Principle 1.3, the same player text in different states must produce different deltas — state-sensitivity is a prompt feature, not a model feature.

### Inputs (four)

1. The current crisis (text describing the situation the player faces).
2. The current state vector (six numeric values, 0–100, from `mechanics/state-axes.md`).
3. The player's free-text policy move.
4. Retrieved wiki context — corpus entries relevant to this crisis and the player's framing.

### Outputs (four)

1. `state_delta` — six numeric changes (typically −20 to +20 per axis).
2. `interpretive_gloss` — 2–4 sentences explaining what the model "heard" the player's move to be.
3. `narrative_move` — 1–2 sentences describing what happens next.
4. `grounding_trace` — list of wiki entry paths the model drew on.
5. `confidence` — `low` | `medium` | `high`.

### Prompt structure

The grammar uses a two-part prompt. The system prompt encodes rules; the user prompt delivers the specific situation.

**System prompt (four sections, ~2,000 tokens):**

1. **Role.** The model is asked to role-play as "the simulation's interpretive layer."
2. **State model.** A compact summary of the six axes, their meanings, the interaction matrix, and the collapse thresholds.
3. **Output schema.** The JSON schema with field descriptions.
4. **Behavior constraints.** Three explicit rules:
   - Produce state-vector deltas that *use* the current state.
   - Ground interpretations in the retrieved wiki context.
   - Do not recommend actions to the player.

**User prompt (five sections, ~4,200 tokens worst case):**

1. **Current crisis.** Generated by the crisis anatomy.
2. **Current state vector.** Six numeric values, named bands, visible signals.
3. **Player's policy move.** Reproduced verbatim, no preprocessing.
4. **Recent move history (last 3 turns).** What the player wrote previously, what deltas resulted.
5. **Retrieved wiki context.** Corpus entries relevant to the crisis and the player's framing.

### State-sensitivity mechanism

Per Principle 1.3, identical player text in different states must produce different deltas. The mechanism has three components:

1. The state vector is in the user prompt.
2. The system prompt forbids state-invariant outputs.
3. The interaction matrix is in the system prompt.

### Wiki retrieval

The grammar uses the parent's `wiki-query.js` pattern. For each turn:

1. Query is built from current crisis + player's policy move.
2. Top 6 pages retrieved from `wiki/index.md`.
3. Pages formatted with title, path, description, content (truncated).
4. Included in user prompt as the final section.

### Interpretive gloss

The gloss is the literacy device's most direct surface. It must:

- Say what the model *heard*, not what the player *meant*.
- Name the state-sensitivity when the same move produces different deltas in different states.
- Surface uncertainty when confidence is low.

### Test cases

Five test cases exercise distinct failure patterns:

1. **Visible-stabilization-hides-hidden-weakening.** A 60-day frontier-model review announcement stabilizes legitimacy but shifts elite_alignment negatively (labs read collaboration as capture).
2. **Upstream conditions but visible signals misread.** A third-party training-data audit is structurally sound but read as "delay" by the public.
3. **Visible crisis addressed but upstream ignored.** A public apology + recall addresses harm but doesn't address systemic cause; narrative coherence erodes.
4. **Collapse path triggered.** An emergency summit in a state where capability has crossed the threshold reads as admission of incapacity.
5. **Policy exceeds reasonable scope.** Criminalizing frontier AI research triggers cascading elite defection and legitimacy loss.

Each test case has crisis, state vector, player move, expected delta (±3 tolerance), expected gloss, and grounding references.
