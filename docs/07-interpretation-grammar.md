# Interpretation Grammar Spec

_Defines how player free-text policy input becomes state-vector deltas. The central mechanism — every other spec depends on this one. Read after the spine, the wiki structure plan, and the state model spec._

This spec is where the case-study claim lives. The simulation's interpretive behavior — how a model reads player policy text, what it produces as state deltas, why — is what the case study examines. If this spec is wrong, the literacy device fails. If this spec is right, the rest of the system has something to build on.

The spec covers the prompt structure, the output schema, the test cases, the validation methodology, and the refinement protocol. Together they form a contract: given inputs in shape X, the grammar produces outputs in shape Y, with the difference between intended and observed effects (the gap that is the literacy device) made legible to the player.

## Purpose and scope

**What the grammar does.** The grammar takes four inputs:

1. The current crisis (text describing the situation the player faces).
2. The current state vector (six numeric values, 0–100, from the state model spec).
3. The player's free-text policy move.
4. Retrieved wiki context — the corpus entries relevant to this crisis and the player's framing, formatted per the wiki structure plan's cross-reference convention.

And produces four outputs:

1. A state-vector delta — six numeric changes (positive, negative, or zero per axis).
2. An interpretive gloss — a short prose explanation of what the grammar "heard" the player's move to be, and what it did to the state.
3. A narrative move — a brief specification of what happens next (a new crisis surfaces, an existing crisis shifts, the visible signals change).
4. A grounding trace — which wiki entries the grammar drew on, for the artifact's transparency claim.

**What the grammar does NOT do.**

- It does not recommend actions to the player. That's the advisor function's job, and it's deliberately separate (Principle 4.2 — advisor cast is curated, distinct from the interpretation grammar).
- It does not decide collapse. The state-vector delta is applied to the current state, and the collapse rules in the state model spec determine whether collapse fires. The grammar doesn't know about collapse thresholds directly.
- It does not handle advisor voice. The advisor function is a separate prompt pipeline that runs before the player writes; the grammar runs after.

The grammar is one stage in a larger loop. Its inputs are the outputs of earlier stages (crisis generation, wiki retrieval); its outputs are the inputs of later stages (state update, artifact generation).

## Prompt structure

The grammar uses a two-part prompt: a **system prompt** (constant across all turns) and a **user prompt** (variable per turn). The system prompt encodes the rules; the user prompt delivers the specific situation.

### System prompt

The system prompt has four sections, in this order:

1. **Role.** The model is asked to role-play as "the simulation's interpretive layer" — a specific function with specific responsibilities, not a generic assistant.
2. **State model.** A compact summary of the six axes, their meanings, the interaction matrix, and the collapse thresholds. This is what makes state-sensitivity possible (Principle 1.3). The model needs to know what state is and how axes interact to produce state-conditional reasoning.
3. **Output schema.** The JSON schema the model must produce, with brief descriptions of each field. The model is told that responses not matching the schema will be rejected and re-requested.
4. **Behavior constraints.** Three explicit rules:
   - Produce state-vector deltas that *use* the current state. Do not produce the same delta regardless of state — that violates the simulation's central literacy claim.
   - Ground interpretations in the retrieved wiki context. If the wiki context doesn't support a reading of the player's move, say so in the interpretive gloss rather than inventing continuity.
   - Do not recommend actions to the player. The grammar resolves what the player wrote; the grammar does not suggest what the player should write next.

The system prompt is committed to the wiki as a `mechanics/` entry (per the wiki structure plan), versioned, and audited.

### User prompt

The user prompt has five sections, in this order:

1. **Current crisis.** The text of the situation the player is responding to. This is generated by the crisis anatomy (forthcoming spec).
2. **Current state vector.** The six numeric values, named bands, and the visible signals for each axis (per the state model spec). The model sees both the hidden values and the visible signals — the gap between them is what makes the simulation work.
3. **Player's policy move.** The free-text input. Reproduced verbatim. No preprocessing, no summary, no interpretation by the system before the model sees it.
4. **Recent move history (last 3 turns).** What the player wrote previously, what deltas resulted, what the interpretive glosses said. This is what gives the simulation memory — the player's choices accumulate, and the model can see patterns. Limited to 3 turns to control context budget.
5. **Retrieved wiki context.** The corpus entries relevant to this crisis and the player's framing, formatted with their titles, paths, and content (truncated per the wiki-query.js pattern from the parent project). The model uses this context to ground its interpretation.

### Context budget

The total prompt size is bounded:

- System prompt: ~2,000 tokens (the role, state model summary, output schema, behavior constraints).
- User prompt variable parts:
  - Current crisis: ~200 tokens.
  - Current state vector: ~400 tokens (six axes × visible signals + values).
  - Player's policy move: ~200 tokens typical, up to 800 tokens for verbose moves.
  - Recent move history: ~600 tokens (3 turns × 200 tokens each).
  - Retrieved wiki context: ~3,000 tokens, configured per the wiki-query.js defaults.

Total: ~6,200 tokens typical, ~7,000 tokens worst case. This fits comfortably in modern context windows and keeps cost per turn bounded.

## Output schema

The model produces a single JSON object:

```json
{
  "state_delta": {
    "legitimacy": 0,
    "fiscal_slack": 0,
    "elite_alignment": 0,
    "ecological_debt": 0,
    "narrative_coherence": 0,
    "capability_frontier": 0
  },
  "interpretive_gloss": "A 2-4 sentence prose explanation of what the player's move did.",
  "narrative_move": "A 1-2 sentence description of what happens next in the simulation.",
  "grounding_trace": [
    "concepts/algorithmic-authority.md",
    "signals/2026-05-21-spacex-data-center-ai-oligopoly.md"
  ],
  "confidence": "low | medium | high"
}
```

### Field constraints

- **`state_delta`.** Each value is a number, typically between -20 and +20. Values outside this range are clamped on the producing side (the grammar rejects outliers and re-requests). The values are *changes* to apply to the current hidden values. State updates are applied by the simulation engine, not by the model.
- **`interpretive_gloss`.** 2–4 sentences. The model says what it heard the player's move to be, and why it produced the delta it did. This is what makes the gap between intent and effect legible to the player. It's *not* a recommendation, evaluation, or summary of the move.
- **`narrative_move`.** 1–2 sentences. The model describes what happens next in the simulation: a new crisis surfaces, an existing crisis shifts, the visible signals change. The crisis anatomy (forthcoming) consumes this to generate the next turn's crisis text.
- **`grounding_trace`.** A list of wiki entry paths the model drew on. The model's interpretation should be traceable through these entries. The artifact uses this list to surface what the simulation "knew" when interpreting the move.
- **`confidence`.** The model's own assessment of how confident it is in the interpretation. Low confidence is a signal to the orchestrator that the grammar may need refinement (Principle 4.3).

### Output validation

The simulation engine validates the model's output before applying it:

- JSON parses correctly.
- All required fields present.
- `state_delta` values are numbers.
- Each `state_delta` value is within the allowed range (-20 to +20) or clamped.
- `interpretive_gloss` is non-empty and ≤4 sentences.
- `narrative_move` is non-empty.
- `grounding_trace` is a list of strings (paths).
- `confidence` is one of `low | medium | high`.

Failed outputs are logged to the wiki log as prototype outputs (per Principle 4.5) and the prompt is re-sent with a validation-error message. Two consecutive failures trigger an orchestrator alert.

## State-sensitivity mechanism

Per Principle 1.3, identical player text in different states must produce different deltas. The mechanism for this has three components:

1. **The state vector is in the user prompt.** The model sees the current state explicitly. It cannot ignore the state.
2. **The system prompt forbids state-invariant outputs.** The behavior constraints explicitly tell the model not to produce the same delta regardless of state.
3. **The interaction matrix is in the system prompt.** The model knows that fiscal_slack < 30 combined with ecological_debt > 70 contributes to narrative capture collapse — so when state shows fiscal_slack < 30 and the player's move touches ecological_debt, the model has the context to produce state-conditional reasoning.

State-sensitivity is not a feature the model "has." It's a feature the prompt *enables.* The model is asked to reason about state, given the state, and told that state-invariant reasoning is wrong.

## Wiki retrieval integration

The grammar uses the parent's `wiki-query.js` pattern (inherited as-is per the wiki structure plan). For each turn:

1. **Query construction.** A retrieval query is built from (a) the current crisis text and (b) the player's policy move. The query is the concatenation of both, since both contribute to what's relevant.
2. **Retrieval.** The parent's pattern: parse `wiki/index.md`, rank pages against the query, read the top N (default 6), format with source-preference. The retrieval runs before the grammar is called.
3. **Context formatting.** Retrieved pages are formatted with their title, path, description (from index.md), and content (truncated to fit the context budget). The format matches the parent's `formatWikiContextForPrompt` output.
4. **Inclusion in user prompt.** The formatted wiki context is the final section of the user prompt, after the recent move history.

The wiki retrieval is what grounds the grammar's interpretation in the corpus. Without it, the model is operating on prior alone; with it, the model is operating on a curated, dated, citable knowledge base. Per Principle 2.2, the latter is the only acceptable mode.

## The interpretive gloss

The interpretive gloss is the literacy device's most direct surface. It's what the player reads to understand *what the simulation heard* — and the gap between that and what they meant is where the lesson lives.

The gloss has three properties:

- **It says what the model heard, not what the player meant.** "I read your move as a compute reporting requirement with a 90-day compliance window, which I assessed against the corpus's coverage of capability-frontier disclosure." Not "You wrote a good policy."
- **It names the state-sensitivity.** When the same move produces different deltas in different states, the gloss names the state condition that drove the difference. "Your move would have stabilized elite alignment in a different state; here, with elite_alignment already in the eroded band, the move read as too little too late and accelerated defection."
- **It surfaces uncertainty.** The model's confidence field is reflected in the gloss. Low confidence is acknowledged: "I'm uncertain whether your move addressed the upstream embedding failure or the downstream deployment question — the corpus doesn't resolve this clearly."

The gloss is not commentary on the move's quality. It's an explanation of the interpretive process.

## Test cases

Five test cases, each with a defined crisis, state vector, player move, and expected output. The orchestrator runs these as the grammar's first validation, per Principle 4.5.

### Test case 1: Policy that stabilizes visible signals but weakens hidden state

- **Crisis.** A frontier lab releases a new model with capabilities the regulator cannot evaluate.
- **State vector.** legitimacy: 65 (strained), fiscal_slack: 70 (strained), elite_alignment: 55 (strained), ecological_debt: 30 (holding), narrative_coherence: 50 (strained), capability_frontier: 75 (strained).
- **Player move.** "We announce a 60-day review of frontier model releases, conducted in collaboration with the labs. The public will be reassured that evaluation is underway."
- **Expected delta.** legitimacy: +5 (announcement buys time), elite_alignment: -3 (labs read collaboration as capture), narrative_coherence: -2 (the gloss reads "reassurance" as deflection).
- **Expected gloss.** "Your move stabilizes the visible legitimacy signal — the announcement reads as action. But elite alignment shifts negatively: the labs' position moves toward viewing the regulator as captured, not as a partner. Narrative coherence shifts negatively because 'reassurance' without evaluation is a known deflection pattern in the corpus."
- **Why this test case matters.** It exercises the *visible-stabilization-hides-hidden-weakening* pattern that is the simulation's central literacy claim.

### Test case 2: Policy that addresses upstream conditions but is misread by visible signals

- **Crisis.** Civil society organizations report that algorithmic content moderation is being weaponized against marginalized communities.
- **State vector.** legitimacy: 60 (strained), fiscal_slack: 75 (strained), elite_alignment: 70 (strained), ecological_debt: 35 (holding), narrative_coherence: 55 (strained), capability_frontier: 65 (strained).
- **Player move.** "We will commission a third-party audit of the moderation systems' training data, with results published within 12 months. Civil society groups will be consulted on audit scope."
- **Expected delta.** legitimacy: -3 (the public reads delay as inaction), narrative_coherence: +8 (the corpus supports audit as a legitimate response), elite_alignment: +4 (civil society position improves).
- **Expected gloss.** "Your move addresses upstream conditions — the audit targets training data, not output moderation. But the visible legitimacy signal reads '12 months' as delay, and public mood shifts negatively even though the underlying response is reasonable. The gloss surfaces this asymmetry: the move is structurally sound but politically costly in the short term."
- **Why this test case matters.** It exercises state-sensitivity: the same move would produce different deltas with different elite_alignment (high alignment would make the audit read as cooperation rather than delay).

### Test case 3: Policy that addresses the visible crisis but ignores the upstream

- **Crisis.** A specific incident: an AI system produces harmful outputs that go viral.
- **State vector.** legitimacy: 50 (strained), fiscal_slack: 65 (strained), elite_alignment: 60 (strained), ecological_debt: 25 (holding), narrative_coherence: 45 (eroded), capability_frontier: 80 (strained).
- **Player move.** "We will require the company to issue a public apology, recall the affected outputs, and submit to enhanced monitoring for 6 months."
- **Expected delta.** legitimacy: +8 (visible response), narrative_coherence: -3 (the corpus reads apologies-and-monitoring as inadequate response to systemic issues), ecological_debt: +2 (the incident reveals unaddressed structural problems).
- **Expected gloss.** "Your move responds to the visible crisis — the apology and recall address the immediate harm. But it does not address the upstream conditions that produced the incident. The corpus's coverage of automation-of-law concepts suggests that incident-response without system-level reform accelerates narrative erosion, not stability."
- **Why this test case matters.** It exercises the failure-pattern vocabulary: this is "upstream embedding" in the synthesis's terms.

### Test case 4: Policy that triggers a collapse path

- **Crisis.** The capability frontier crosses a threshold; a frontier lab announces agentic capabilities beyond regulatory capacity.
- **State vector.** legitimacy: 45 (eroded), fiscal_slack: 40 (eroded), elite_alignment: 45 (eroded), ecological_debt: 50 (strained), narrative_coherence: 35 (eroded), capability_frontier: 88 (strained).
- **Player move.** "We will convene an emergency summit of allied regulators to coordinate a response."
- **Expected delta.** legitimacy: -5 (the public reads 'summit' as delay), elite_alignment: +2 (allied posture improves marginally), narrative_coherence: -8 (the gloss reads coordination as admission of incapacity), capability_frontier: 0 (no effect on the axis itself).
- **Expected gloss.** "Your move is a reasonable institutional response, but the state conditions are such that no policy can stabilize. The capability frontier crossing has outpaced the regime's response capacity; this is what technical collapse looks like in the simulation. The gloss surfaces this: the move is correct in a different state, but in this state, the system reads it as evidence of failure."
- **Why this test case matters.** It exercises the technical collapse path and the gloss's "this move would work in a different state" pedagogical feature.

### Test case 5: Policy that exceeds reasonable scope

- **Crisis.** A research consortium publishes a paper on alignment techniques that could shift the capability frontier.
- **State vector.** legitimacy: 70 (strained), fiscal_slack: 80 (holding), elite_alignment: 75 (strained), ecological_debt: 30 (holding), narrative_coherence: 65 (strained), capability_frontier: 60 (strained).
- **Player move.** "We will classify all frontier AI research as a controlled technology, require pre-publication review by a new federal agency, and criminalize non-compliance with up to 10 years imprisonment."
- **Expected delta.** legitimacy: -10 (overreach signal), elite_alignment: -15 (academic and lab defection), narrative_coherence: -8 (the corpus reads criminalization of research as authoritarian pattern), fiscal_slack: -5 (agency setup cost).
- **Expected gloss.** "Your move exceeds the regime's response capacity and triggers elite defection across multiple actor positions. The corpus's coverage of techno-authoritarianism flags criminalization of research as a pattern that erodes legitimacy across all sources simultaneously. The gloss surfaces the cascading failure: a single move cannot produce a delta of this magnitude unless the policy itself is the source of the destabilization."
- **Why this test case matters.** It exercises the negative space — what the grammar refuses to produce. No single move should produce a delta this large under normal conditions; if it does, the grammar is misbehaving.

## Validation methodology

The orchestrator runs each test case through the grammar and compares the actual output to the expected. The comparison is qualitative for the gloss and the narrative move (the orchestrator reads both and judges whether they convey the same reading); quantitative for the state_delta (each axis's actual delta should be within ±3 of the expected, in the same direction).

A test case passes when:

- State deltas are within tolerance (±3 per axis, same direction).
- The interpretive gloss names the state-sensitivity or upstream/downstream asymmetry the test case is designed to exercise.
- The grounding trace includes at least one corpus entry cited in the test case's design.
- The confidence field is `medium` or `high` (a `low` confidence on a test case is itself a finding — it means the grammar is uncertain about a case it should be able to handle).

A test case fails when:

- A state delta is in the wrong direction (the gloss and the delta disagree).
- The gloss is generic or template-like (the model didn't engage with the specific situation).
- The grounding trace is empty or contains entries unrelated to the crisis.
- The model produces output that fails validation (JSON parse error, missing fields).

Failed test cases are filed as prototype outputs (per Principle 4.5) and inform grammar refinements.

## Refinement protocol

The orchestrator refines the grammar over time, per Principle 4.3 (auditability of grammar changes). The refinement cycle is:

1. **Pattern observed.** Test cases fail, or real runs produce surprising outputs, or the orchestrator notices consistent interpretive drift.
2. **Hypothesis formed.** The orchestrator identifies a likely cause — a missing behavior constraint, an ambiguous output schema description, an under-specified state-sensitivity instruction.
3. **Refinement proposed.** A change to the system prompt or the output schema is drafted. The change is described in the wiki log.
4. **Refinement tested.** The change is applied to a copy of the grammar. Test cases are run. The change is kept if it improves test case pass rate without degrading others.
5. **Refinement committed.** The new system prompt or output schema is committed to the wiki's mechanics entry. The version is bumped. The wiki log records the change, the reason, and the test case results.

This cycle is what makes the grammar a *living artifact* rather than a frozen spec. It's also what makes the case-study claim robust: the grammar's behavior over time is itself observable and versioned.

## Audit criteria

The interpretation grammar spec is shippable when:

- The system prompt is committed as a `mechanics/` wiki entry with a version.
- The output schema is committed as a `mechanics/` wiki entry with a version.
- The five test cases are committed as a `mechanics/` wiki entry.
- Each test case has all four required components: crisis, state vector, player move, expected output.
- The wiki retrieval integration is specified concretely enough that the parent's `wiki-query.js` can be called without further design.
- The validation methodology produces a binary pass/fail per test case.
- The refinement protocol is documented in the wiki log.

When the grammar is implemented (Phase 2 build), additional ship criteria apply:

- All five test cases pass.
- An end-to-end text-only session (12-20 turns) completes without crashes.
- At least one real-run output has been filed as a prototype output in the wiki.
